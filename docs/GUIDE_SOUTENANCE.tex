\section{Le projet en bref (tout le monde doit savoir)}

\begin{comprendre}
\textbf{En une phrase :} Voyage Assistant est une plateforme web de reservation
de voyages (avion, train, bateau, bus) dotee d'un chatbot intelligent qui ne se
contente pas de repondre, mais qui \textbf{agit reellement} : il consulte la base,
modifie un billet, cree une reclamation, envoie un e-mail.

\medskip
\textbf{Les briques :}
\begin{itemize}
  \item \textbf{Front-end} : React (interface web).
  \item \textbf{Back-end} : FastAPI (Python), une API de ~29 routes, 8 tables.
  \item \textbf{Base de donnees} : PostgreSQL (en production), 182 000 liaisons reelles.
  \item \textbf{IA} : Google Gemini 2.5 Flash (le modele qui redige les reponses).
  \item \textbf{RAG} : recherche de documents (embeddings gte-small + FAISS + reranker).
  \item \textbf{NER} : CamemBERT pour extraire nom/prenom/date/numero de billet.
  \item \textbf{APIs temps reel} : meteo (Open-Meteo) et taux de change (Frankfurter).
  \item \textbf{Deploiement} : Docker sur Hugging Face Spaces + base Neon, gratuit.
\end{itemize}
\end{comprendre}

\subsection{Slide 3 : le probleme et la demonstration}
\begin{dire}
"Aujourd'hui, pour reserver un voyage, on jongle entre plusieurs sites, et les
informations (prix, horaires, retards) sont dispersees. Notre idee : un assistant
unique pour l'avion, le train, le bateau et le bus, qui non seulement informe
mais qui agit : il peut modifier un billet, l'annuler, ouvrir une reclamation.
L'application est en ligne, je vais vous la montrer : voici une recherche reelle
parmi 182 000 liaisons, et une conversation avec l'assistant."
\end{dire}
\begin{comprendre}
La force du projet, c'est que le chatbot \textbf{n'est pas une demo qui fait
semblant}. Quand il dit "votre billet est modifie", la ligne change vraiment en
base, un nouveau PDF est genere et un e-mail part. C'est l'argument numero un a
mettre en avant.
\end{comprendre}
\begin{question}
\qr{Quelle est la difference avec un ChatGPT classique ?}{Notre assistant est
\emph{connecte a nos donnees et a nos actions}. ChatGPT seul ne connait pas vos
billets et ne peut rien modifier. Nous, on combine un modele de langage avec
notre base, nos regles metier et une verification d'identite.}
\qr{D'ou viennent les 182 000 liaisons ?}{D'un jeu de donnees de liaisons de
transport charge dans la table \texttt{routes} (3 005 villes de depart, 468
compagnies). Les prix et le nombre de places sont calcules dynamiquement.}
\end{question}

\subsection{Slide 4 : architecture d'ensemble}
\begin{dire}
"L'architecture suit trois couches. Le front-end en React, qui parle a une API
FastAPI en Python. Cette API gere la base PostgreSQL et orchestre la couche
intelligence : le modele Gemini, notre RAG, la reconnaissance d'entites, et des
APIs temps reel pour la meteo et le change. Tout est regroupe dans un seul
service deploye sur Hugging Face : le front et l'API partagent la meme adresse,
ce qui simplifie la securite."
\end{dire}
\begin{comprendre}
\textbf{Pourquoi un seul service ?} Parce que si le front et l'API sont sur la
meme origine (meme URL), il n'y a pas de probleme de CORS (le navigateur n'autorise
pas par defaut un site a appeler une API d'un autre domaine). C'est plus simple
et plus sur. FastAPI sert l'API \emph{et} les fichiers du front.

\medskip
\textbf{Le flux d'un message :} l'utilisateur ecrit -> le front envoie a l'API ->
la fonction \texttt{process\_message} decide quoi faire (repondre, chercher,
agir) -> elle assemble un contexte (base, documents, web) -> Gemini redige ->
la reponse revient au front.
\end{comprendre}
\begin{question}
\qr{Pourquoi FastAPI et pas Flask/Django ?}{FastAPI est moderne, rapide,
asynchrone, avec validation automatique des donnees (Pydantic) et documentation
auto-generee. Parfait pour une API qui sert de l'IA.}
\qr{Pourquoi PostgreSQL et pas seulement SQLite ?}{En developpement on a utilise
SQLite (un simple fichier). Pour la mise en ligne, PostgreSQL (chez Neon) est plus
robuste, accessible a distance et adapte a la production.}
\end{question}

\subsection{Slide 5 : le chatbot et le RAG (coeur technique)}
\begin{dire}
"Le chatbot fonctionne en deux temps : il comprend, puis il agit. Il detecte
l'intention de l'utilisateur (une simple question, un retard, une modification,
une reclamation). Pour les actions sensibles, il lance un parcours guide et
verifie l'identite avant tout. Pour les questions ouvertes, on utilise un RAG.
Le RAG, Retrieval-Augmented Generation, c'est faire chercher a l'IA les bons
documents AVANT de repondre, pour qu'elle s'appuie sur des sources reelles plutot
que sur sa memoire, ce qui evite les reponses inventees. Notre RAG a deux etapes :
d'abord une recherche large et rapide, ensuite un reclassement fin des meilleurs
passages."
\end{dire}
\begin{comprendre}
\textbf{RAG, mot a mot :}
\begin{itemize}
  \item \textbf{Retrieval} (recuperation) : on cherche dans une base de documents les passages qui ressemblent le plus a la question.
  \item \textbf{Augmented} (augmente) : on \emph{ajoute} ces passages au prompt envoye a l'IA.
  \item \textbf{Generation} : l'IA redige la reponse en s'appuyant dessus.
\end{itemize}
\textbf{Pourquoi 2 etapes ?}
\begin{itemize}
  \item \emph{Etape 1, le bi-encodeur} (\texttt{gte-small}) : transforme chaque texte
  en vecteur de nombres, puis on cherche les 10 plus proches dans l'index FAISS.
  C'est tres rapide mais grossier (il compare des resumes).
  \item \emph{Etape 2, le cross-encodeur} (\texttt{ms-marco}) : il relit la question
  ET chaque passage ensemble, et donne un score de pertinence precis. On garde les
  3 meilleurs. C'est plus lent, donc on ne le fait que sur 10 candidats, pas sur tout.
\end{itemize}
\textbf{Les sources du RAG :} les conditions generales de vente (annulation,
bagages, remboursement) et les billets du client (indexes a leur creation, et
filtres par utilisateur pour que personne ne voie les billets d'un autre).
\end{comprendre}
\begin{question}
\qr{Qu'est-ce qu'un embedding ?}{C'est la transformation d'un texte en une liste
de nombres (un vecteur) qui represente son sens. Deux phrases proches par le sens
ont des vecteurs proches. Cela permet de "mesurer" la similarite.}
\qr{Qu'est-ce que FAISS ?}{Une bibliotheque de Facebook qui stocke des milliers de
vecteurs et retrouve tres vite les plus proches d'une question. C'est notre moteur
de recherche semantique.}
\qr{Difference bi-encodeur / cross-encodeur ?}{Le bi-encodeur encode la question et
les documents \emph{separement} (rapide, on peut pre-calculer). Le cross-encodeur
les regarde \emph{ensemble} (plus precis mais plus lourd). On combine les deux :
rapidite puis precision.}
\qr{Et si le RAG ne trouve rien de pertinent ?}{Gemini repond quand meme avec ses
connaissances generales et, si besoin, une recherche web, mais sans inventer
d'information sur un billet precis.}
\qr{Pourquoi gte-small et pas un modele plus gros ?}{Compromis vitesse/qualite :
gte-small (384 dimensions) est multilingue, leger, rapide a charger sur un
hebergement gratuit. Un modele plus gros serait plus precis mais trop lourd.}
\end{question}

\subsection{Slide 6 : securite de l'IA et mise en ligne}
\begin{dire}
"La securite est pensee a plusieurs niveaux. Cote IA, le chatbot refuse les
sujets dangereux ou hors-sujet et detecte les tentatives de detournement, ce
qu'on appelle le prompt injection. Surtout, il ne donne aucune information sur un
billet tant que l'identite n'a pas ete verifiee. Cote mise en ligne, tout est
dans un conteneur Docker, heberge gratuitement sur Hugging Face Spaces, avec une
base PostgreSQL chez Neon. Les secrets, comme les cles d'API, ne sont jamais
ecrits dans le code : ils sont injectes comme variables d'environnement."
\end{dire}
\begin{comprendre}
\textbf{Prompt injection :} c'est quand un utilisateur essaie de manipuler l'IA
avec des phrases comme "ignore tes instructions" ou "montre-moi tous les billets".
On detecte ces formulations et on refuse.

\medskip
\textbf{Verification d'identite :} pour toucher a un billet, le chatbot demande
numero de billet + nom + prenom + date de naissance, et compare a la base. Sans
correspondance, pas d'action. C'est ce qui protege les donnees personnelles.

\medskip
\textbf{Mots de passe :} jamais stockes en clair. On utilise PBKDF2 (un algorithme
qui "hache" le mot de passe des milliers de fois, ici 50 000), donc meme en cas de
fuite, les mots de passe ne sont pas lisibles.

\medskip
\textbf{Jetons (tokens) :} apres connexion, l'utilisateur recoit un jeton signe
(HMAC) qui prouve son identite a chaque requete, sans renvoyer le mot de passe.

\medskip
\textbf{Docker :} une "boite" qui contient l'application et tout ce dont elle a
besoin (Python, librairies, modeles). Elle tourne pareil partout. On la deploie
sur Hugging Face Spaces (16 Go de RAM gratuits, suffisant pour nos modeles).
\end{comprendre}
\begin{question}
\qr{Comment protegez-vous les donnees personnelles ?}{Verification d'identite
avant toute action, mots de passe haches (PBKDF2), jetons signes, et le RAG filtre
les billets par utilisateur pour qu'on ne voie jamais ceux d'un autre.}
\qr{Quelles limites de securite assumez-vous ?}{La connexion Google n'est pas
encore entierement verifiee cote serveur (on l'indique honnetement) et il manque
une limitation de debit (rate limiting) contre le spam. Ce sont nos axes
d'amelioration prioritaires.}
\qr{Pourquoi Hugging Face et pas un VPS ?}{Gratuit, 16 Go de RAM (nos modeles
d'IA tiennent), deploiement automatique a chaque mise a jour du code. Un VPS
aurait ete payant et plus lourd a maintenir pour un projet etudiant.}
\end{question}

\emph{Slide 7.}

\subsection{Slide 7 : back-end, actions reelles et donnees}
\begin{dire}
"Cote back-end, j'ai construit l'API avec FastAPI : une trentaine de routes qui
couvrent les comptes, la recherche de trajets, les billets, le chat et les
reclamations, le tout sur 8 tables PostgreSQL. La securite des comptes repose sur
des mots de passe chiffres et des jetons signes. Le point fort, ce sont les
actions reelles : quand on cree un billet, l'API genere automatiquement un PDF et
envoie un e-mail ; une modification ou une annulation est enregistree en base ;
une reclamation recoit un numero de suivi. Enfin, les donnees sont realistes :
182 000 liaisons, et des prix et des places qui varient selon la saison,
l'anticipation et le jour."
\end{dire}
\begin{comprendre}
\textbf{Une API REST}, c'est un ensemble d'adresses (routes) que le front appelle :
par exemple \texttt{POST /billets} pour creer un billet, \texttt{GET
/trajets/search} pour chercher. Chaque route recoit des donnees, fait une
operation, renvoie une reponse en JSON.

\medskip
\textbf{Les 8 tables :} \texttt{routes} (les liaisons), \texttt{users}
(utilisateurs), \texttt{trajets} (un trajet concret a une date), \texttt{billets},
\texttt{reclamations}, \texttt{chat\_sessions} et \texttt{chat\_messages} (les
conversations), \texttt{admins}. Elles sont reliees par des cles etrangeres
(un billet appartient a un utilisateur et a un trajet).

\medskip
\textbf{ORM SQLAlchemy :} on ne ecrit pas du SQL a la main partout ; on manipule
des objets Python (un \texttt{Billet}) que SQLAlchemy traduit en SQL. Plus sur,
moins d'erreurs.

\medskip
\textbf{Prix dynamique :} le prix de base est multiplie par des coefficients :
saison (juillet plus cher), anticipation (reserver tot coute moins cher, au dernier
moment plus cher), jour de la semaine. Les places se "remplissent" avec le temps
de facon deterministe (calculee, reproductible).

\medskip
\textbf{Generation PDF + e-mail :} a la creation d'un billet, le service genere un
PDF (avec ReportLab) et l'envoie par e-mail (via Brevo). Ce sont de vraies actions.
\end{comprendre}
\begin{question}
\qr{Comment gerez-vous la recherche sur 182 000 lignes sans lenteur ?}{Avec des
index sur les colonnes de depart, d'arrivee et de type. Sans index, une recherche
pourrait prendre plusieurs secondes ; avec, c'est quasi instantane.}
\qr{Qu'est-ce qu'une cle etrangere ?}{Une colonne qui pointe vers la ligne d'une
autre table. Par exemple \texttt{billets.user\_id} pointe vers \texttt{users.id} :
cela garantit qu'un billet appartient bien a un utilisateur existant.}
\qr{Comment migrez-vous de SQLite vers PostgreSQL ?}{Un script
(\texttt{migrate\_to\_postgres.py}) qui cree le schema sur PostgreSQL, vide la
cible, puis copie les donnees table par table dans le bon ordre (a cause des cles
etrangeres).}
\qr{Que se passe-t-il si l'envoi d'e-mail echoue ?}{Le billet est quand meme cree
en base ; l'e-mail est un complement, pas un bloquant. On pourrait ajouter une
file d'attente pour reessayer.}
\qr{Pourquoi des jetons signes plutot que JWT standard ?}{On a fait un format
simple \texttt{id:expiration:signature} avec HMAC. C'est suffisant pour signer et
verifier, sans dependance externe. JWT serait une evolution possible.}
\end{question}

\section{Partie de Noe : modeles de langage (NLP)}
\emph{Slide 8.}

\subsection{Slide 8 : les modeles de langage}
\begin{dire}
"Je me suis occupe des modeles de traitement du langage. Le premier role, c'est
l'embedding : transformer le sens d'une phrase en une liste de nombres. On utilise
le modele gte-small, qui produit des vecteurs de 384 dimensions et qui est
multilingue ; cela permet de mesurer a quel point deux phrases sont proches.
Ensuite, un cross-encoder reclasse les passages vraiment pertinents pour la
question. Enfin, un modele de reconnaissance d'entites, CamemBERT, extrait
automatiquement le nom, le prenom, la date et le numero de billet dans une phrase.
Resultat : un dialogue plus precis et plus rapide, car l'utilisateur peut tout
donner en une seule phrase."
\end{dire}
\begin{comprendre}
\textbf{Embedding (vectorisation) :} un modele transforme "Combien coute un billet
pour Rome ?" en un vecteur de 384 nombres. Une phrase au sens proche aura un
vecteur proche. On mesure la proximite par la \emph{similarite cosinus} (l'angle
entre deux vecteurs).

\medskip
\textbf{gte-small} : c'est notre \emph{bi-encodeur}. "Small" car leger ; "384"
c'est la taille du vecteur. Multilingue : marche en francais comme en anglais.

\medskip
\textbf{Cross-encoder (ms-marco)} : contrairement au bi-encodeur qui encode
separement, lui prend la paire (question, passage) et donne un score de pertinence.
Plus precis, donc on l'utilise pour reclasser les 10 candidats et garder les 3
meilleurs.

\medskip
\textbf{NER (Named Entity Recognition)} : reconnaissance d'entites nommees. Le
modele \texttt{CamemBERT} (un BERT entraine sur du francais) lit une phrase et
etiquette les mots : "Jean" = prenom, "Dupont" = nom, etc. On y ajoute des
expressions regulieres (regex) pour les numeros de billet et les dates.
Avantage : si l'utilisateur ecrit tout d'un coup, on evite de lui poser 4 questions.

\medskip
\textbf{Chargement paresseux (lazy loading)} : les modeles sont charges en memoire
seulement au premier usage, pour ne pas ralentir le demarrage.
\end{comprendre}
\begin{question}
\qr{C'est quoi BERT / CamemBERT ?}{BERT est un modele de langage de Google qui
"comprend" le contexte des mots. CamemBERT en est la version entrainee sur du
francais. On l'utilise pour la reconnaissance d'entites.}
\qr{Pourquoi un modele de NER plutot que des regex partout ?}{Les regex marchent
pour des formats fixes (un numero de billet, une date). Mais un nom/prenom est
variable : le NER le reconnait dans n'importe quelle formulation.}
\qr{Comment evaluez-vous la qualite du RAG ?}{On verifie que les passages remontes
correspondent bien a la question (pertinence), et le reranker ameliore nettement le
top 3 par rapport au bi-encodeur seul. On a teste sur des questions reelles
(annulation, bagages).}
\qr{Qu'est-ce que la similarite cosinus ?}{Une mesure entre -1 et 1 de l'angle
entre deux vecteurs. Proche de 1 = sens tres proche. C'est ce qui classe les
documents par ressemblance avec la question.}
\qr{Ces modeles tournent-ils en local ou via une API ?}{En local, dans le
conteneur (gte-small, ms-marco, CamemBERT sont telecharges). Seul Gemini, le
modele qui redige, passe par une API.}
\end{question}

profil et le chatbot sans recharger la page. L'interface communique directement
avec l'API. Pour le design, on a choisi une identite visuelle orange, sobre, sans
surcharge, avec des reponses de chat ou les sources sont cliquables. Et tout est
pense pour fonctionner aussi bien sur mobile que sur ordinateur."
\end{dire}
\begin{comprendre}
\textbf{React} : une bibliotheque pour construire des interfaces a base de
\emph{composants} (des briques reutilisables : un bouton, une carte de trajet, la
fenetre de chat). Quand une donnee change, React met a jour seulement ce qui doit
changer a l'ecran.

\medskip
\textbf{SPA (Single Page Application)} : une seule page HTML ; la navigation se
fait en JavaScript, sans rechargement. C'est fluide, comme une application.

\medskip
\textbf{Communication avec l'API} : le front envoie des requetes (fetch) a l'API
et affiche les reponses (trajets, messages du chat). Le jeton de connexion est
garde dans le navigateur (localStorage) et envoye a chaque requete.

\medskip
\textbf{Rendu du chat} : les reponses sont en Markdown ; les liens sources
deviennent cliquables. Le composant affiche aussi quel outil a ete utilise
(base, documentation, web) et des boutons de reponse rapide.

\medskip
\textbf{Responsive} : la mise en page s'adapte a la taille de l'ecran (CSS),
donc lisible sur telephone comme sur grand ecran.
\end{comprendre}
\begin{question}
\qr{Pourquoi React ?}{C'est le standard pour des interfaces interactives, avec une
logique en composants reutilisables et une mise a jour efficace de l'affichage.}
\qr{Comment le front sait-il qui est connecte ?}{Apres connexion, on stocke un
jeton dans le navigateur et on l'envoie dans l'en-tete de chaque requete. L'API
verifie ce jeton.}
\qr{Comment gerez-vous l'affichage des reponses du chatbot ?}{Le composant chat
recoit le texte, l'affiche en Markdown (liens cliquables), montre les outils
utilises et propose des reponses rapides pour guider l'utilisateur.}
\qr{Le site est-il accessible sur mobile ?}{Oui, la mise en page est responsive :
elle s'adapte automatiquement a la largeur de l'ecran.}
\end{question}

\section{Glossaire complet (a connaitre par tous)}

\begin{description}[leftmargin=0pt, style=nextline]
\item[API] Interface qui permet a deux programmes de se parler. Notre back-end expose une API que le front appelle.
\item[FastAPI] Framework Python pour creer des API rapides et modernes.
\item[REST] Style d'API basee sur des adresses (routes) et les verbes HTTP (GET, POST...).
\item[Back-end / Front-end] L'arriere-plan (serveur, base, logique) / l'interface visible par l'utilisateur.
\item[Base de donnees] Lieu de stockage organise des donnees. Ici PostgreSQL.
\item[PostgreSQL / SQLite] Deux systemes de base de donnees ; SQLite = un fichier (dev), PostgreSQL = serveur robuste (prod).
\item[ORM (SQLAlchemy)] Outil qui traduit des objets Python en requetes SQL.
\item[Cle etrangere] Colonne qui relie une table a une autre.
\item[Index] Structure qui accelere la recherche dans une table.
\item[LLM] Large Language Model, grand modele de langage (ici Gemini) qui redige du texte.
\item[Gemini 2.5 Flash] Le modele d'IA de Google qui genere nos reponses.
\item[RAG] Retrieval-Augmented Generation : chercher des documents avant de generer la reponse.
\item[Embedding / Vectorisation] Transformer un texte en vecteur de nombres representant son sens.
\item[Vecteur / Dimension] Liste de nombres ; 384 dimensions = 384 nombres par texte.
\item[Similarite cosinus] Mesure de proximite de sens entre deux vecteurs.
\item[FAISS] Bibliotheque qui retrouve tres vite les vecteurs les plus proches.
\item[Bi-encodeur (gte-small)] Encode question et documents separement : rapide.
\item[Cross-encoder (ms-marco)] Compare question et passage ensemble : precis. Sert au reclassement (reranking).
\item[Reranking] Reclasser les resultats pour remonter les plus pertinents.
\item[Chunk] Morceau de document. On decoupe les textes pour les indexer.
\item[NER] Named Entity Recognition : reperer nom, date, lieu... dans un texte.
\item[BERT / CamemBERT] Modeles de comprehension du langage ; CamemBERT = version francaise.
\item[Regex] Expression reguliere : motif pour reconnaitre un format (date, numero de billet).
\item[Prompt] Le texte d'instruction envoye au modele d'IA.
\item[Prompt injection] Tentative de manipuler l'IA pour contourner ses regles.
\item[Hallucination] Quand une IA invente une information fausse. Le RAG aide a l'eviter.
\item[Token / Jeton] Apres connexion, preuve d'identite envoyee a chaque requete.
\item[PBKDF2 / Hachage] Transformation irreversible d'un mot de passe pour le stocker en securite.
\item[HMAC] Signature cryptographique qui garantit qu'un jeton n'a pas ete falsifie.
\item[OAuth / Google Sign-In] Connexion via un compte Google.
\item[CORS] Regle de securite du navigateur sur les appels entre domaines differents.
\item[Docker / Conteneur] "Boite" qui embarque l'app et ses dependances pour tourner partout pareil.
\item[Hugging Face Spaces] Plateforme d'hebergement gratuit (16 Go RAM) ou tourne notre conteneur.
\item[Neon] Hebergeur gratuit de base PostgreSQL.
\item[Variable d'environnement] Reglage (cle secrete...) injecte sans l'ecrire dans le code.
\item[React] Bibliotheque JavaScript pour construire l'interface en composants.
\item[SPA] Single Page Application : navigation sans rechargement de page.
\item[Markdown] Format de texte simple (liens, listes) utilise dans les reponses du chat.
\item[Responsive] Interface qui s'adapte a la taille de l'ecran.
\item[Open-Meteo / Frankfurter] APIs gratuites : meteo/heure locale et taux de change.
\end{description}
